## Overview

Part 11 is the final training sweep of this project. It combines everything collected across Parts 10 and 11: 8 external signal caches, 6 calendar-rule features replacing a leaky anomaly flag, and a lifecycle expansion from 14 to 21 days with 7 new interaction terms. The result is a model with roughly 58 features (up from ~30 in Part 7) evaluated on the final 3 weeks of each event's selling timeline.

**Result: 13.37 RMSE on the final 3-week test window** (vs. 18.53 in the Part 7 champion, evaluated on all test events).

> **Evaluation window caveat:** Part 7's 18.53 RMSE covers all 58,022 test events, including the majority that have zero sales where predicting zero is easy. Part 11's 13.37 covers only the final 3-week window (389,230 rows from events with days_to_event ≤ 21). These are different populations, so the numbers are not directly comparable. The final 3-week window is the business-relevant period: it is when pricing decisions matter most, but a fair head-to-head would require re-evaluating Part 7 on the same subset.

This post covers the data leakage fix, the full signal join architecture, the lifecycle interaction that drove most of the improvement, results by segment, and what the feature importance analysis reveals.

## Data Leakage Audit

### What Was Wrong

The Part 7 model included an `is_anomaly_day` feature, a binary flag marking specific snapshot dates as anomalous. These dates were identified by running z-score outlier detection on daily sales aggregates across the **full dataset including the test period** (Jan 2026). Below is an updated anomaly chart. 

<img width="1388" height="990" alt="image" src="https://github.com/user-attachments/assets/5e03aa71-d4cc-489d-9691-11919cfff163" />

*Figure 1: Anomaly-sale dates in my data*


Seven dates were flagged as anomalies:

| Date | Train or Test | Reason |
|------|--------------|--------|
| 2025-10-21 | Train | NBA season opener |
| 2025-11-04 | Train | Election Day |
| 2025-12-13 | Train | Holiday shopping surge |
| 2026-01-06 | **Test** | Epiphany week (post-holiday slump) |
| 2026-01-16 | **Test** | MLK weekend |
| 2026-01-17 | **Test** | MLK weekend |
| 2026-01-30 | **Test** | Pre-Super Bowl |

Four of the seven flagged dates fall in the test period. Hardcoding those as features means the model receives a hint! "This is an unusual day", which was derived from observing test-set sales patterns. That is meta-leakage: the flag is not derived from a single row's features, but from aggregate test-set behavior that should not be visible at training time.

### Fixing with Calendar Rules

Each anomaly date has a straightforward calendar explanation that can be computed from `snapshot_date` alone, no sales data needed. I replaced `is_anomaly_day` with six calendar-rule features:

| Feature | Logic |
|---------|-------|
| `is_epiphany_week` | January 1–8: post-holiday market freeze |
| `is_mlk_weekend` | 3rd Monday of January ± 3 days |
| `is_super_bowl_week` | 7–10 days before Super Bowl Sunday |
| `is_election_day` | 1st Tuesday after 1st Monday in November (even years only) |
| `is_holiday_shopping_surge` | December 13–24 |
| `is_season_opener_week` | October 21–28 (NBA season start) |

These features are fully derivable at prediction time and generalize to any future year. The Super Bowl window was extended to 10 days (from 7) after checking that Jan 30 (the closest test-period anomaly date) is 10 days before Super Bowl Sunday on Feb 9.

All 7 anomaly dates are now covered by one of the six calendar rules, with no reference to test-set sales.

> **Outcome:** Calendar rule features still registered **zero importance** in the trained model. The flags cover only a handful of dates across 7.5M rows, which is too sparse for a tree model to find a useful split. They serve as documentation of the leakage fix but do not improve accuracy. `is_anomaly_day` was removed. No replacement feature added net value.

## Full Signal Join Architecture

Eight external signal sources were collected, uploaded to BigQuery, and joined onto the main training table (`mart_event_snapshot_panel`). Each source uses a different join key.

<img width="7370" height="5805" alt="Br_ Cache Flow-2026-03-10-143903" src="https://github.com/user-attachments/assets/9ad814d2-70cc-468e-9c38-5c85f3eb93e9" />

*Figure 2: Joining all the newly collected data*


**Join key summary:**

| Signal | Source | Join Key | Type |
|--------|--------|----------|------|
| Last.fm listeners | Last.fm API | `slug` | Static |
| Spotify popularity/followers | Spotify Web API | `slug` | Static |
| News sentiment (tone, volume) | GDELT Project | `slug + snapshot_date` | Daily, PIT |
| Wikipedia hype (pageviews) | Wikimedia REST API | `slug + snapshot_date` | Daily, PIT |
| Team records (W/L/streak) | nba_api, nfl_data_py, MLB StatsAPI | `slug + snapshot_date` | Daily, PIT |
| Inventory velocity (burn rate) | Derived from listings_active diff | `event_id + snapshot_date` | Daily, PIT |
| Weather (temp, precip, snow) | Open-Meteo | `venue_name + event_date` | Per event date |
| Holiday proximity | Derived from snapshot_date | `snapshot_date` | Calendar lookup |

**Point-in-Time integrity:** All daily-keyed signals are aggregated only up to `snapshot_date`. A sports record joined for a January 10 snapshot reflects the team's record through January 9, not the end-of-season result.

## Changing to a Final 3-Week Window

The Part 7 model used `is_final_week` (days_to_event ≤ 7) as a flag for the critical selling window. Part 11 extends this to 21 days and adds segment-specific interactions.

**New features added:**

```python
df["is_final_3_weeks"] = (df["days_to_event"] <= 21).astype(int)

# 7 interaction terms (one per focus bucket)
for bucket in ["major_sports", "minor_sports", "concert", "festivals",
               "broadway_theater", "comedy", "other"]:
    df[f"final_3_weeks_{bucket}"] = (
        df["is_final_3_weeks"] * df[f"bucket_{bucket}"]
    )
```

The hypothesis was that different event categories behave very differently in the final 3 weeks. A Major Sports game 21 days out has a demand profile that looks nothing like a Broadway show 21 days out. An interaction term like `is_final_3_weeks × is_major_sports` gives the model an explicit feature to learn that difference. Without it, the model has to learn the interaction implicitly from the data, which requires more examples and more model capacity.

This turned out to be the single most important change in the entire Part 11 sweep.

## Training Setup

**Data:**
- Training set: `snapshot_date < 2026-01-01` (Oct–Dec 2025)
- Test set: `snapshot_date ≥ 2026-01-01`, restricted to `days_to_event ≤ 21` (final 3-week window)
- Test rows: 389,230

**Models trained:**
- **Classifier:** XGBoost vs MLP (predicts whether any sales will occur, binary)
- **Regressor:** XGBoost vs MLP (predicts sales volume on positive cases)
- All combinations run with Optuna hyperparameter search, yet again

**Evaluation metric:**
- Classifier: PR-AUC (precision-recall for my data's zero-sales sparsity)
- Regressor: log-RMSE on positive rows
- End-to-end RMSE in ticket units on the held-out test set

### End-to-End RMSE: 13.37 tickets at threshold = 0.60

### Threshold Stability

One notable improvement over Part 7: the threshold curve is now flat.

<img width="1000" height="600" alt="image" src="https://github.com/user-attachments/assets/7dc6ff39-67bd-4462-9a6a-9785ffa24141" />

*Figure 3: Threshold optimization curve*


In Part 7, RMSE dropped substantially as the classifier threshold increased from 0.50 to 0.90. The model needed a high threshold (0.90) to suppress false positives from the classifier. The optimal threshold was sensitive to the exact configuration.

In Part 11, RMSE is essentially constant from threshold 0.10 to 0.65 (all within 13.37–13.41). The classifier is better calibrated and it does not need a high threshold to produce clean predictions. Optimal threshold: **0.60**.

This is a sign of a more robust classifier, not just a better-tuned one.

## Feature Importance: What the Model Actually Used

The top features by importance in the trained model:

| Feature | Importance | Type |
|---------|-----------|------|
| `final_3_weeks_major_sports` | **0.4813** | Lifecycle interaction (new) |
| `is_final_3_weeks` | 0.0760 | Lifecycle flag (new) |
| `listings_active_log` | 0.0348 | Core market |
| `focus_bucket_broadway_and_theater` | 0.0287 | Segment |
| `focus_bucket_comedy` | 0.0280 | Segment |
| `venue_capacity` | 0.0219 | Core market |
| `spotify_popularity` | **0.0206** | External signal (new) |
| `final_3_weeks_concerts` | 0.0164 | Lifecycle interaction (new) |
| `lastfm_listeners` | 0.0097 | External signal (new) |

### The Dominant Discovery

`final_3_weeks_major_sports` accounts for 48% of total feature importance, nearly 5 times the second-most-important feature. This was the central finding of the sweep.

The interaction captures a pattern the model previously had no direct way to express: **Major Sports events in the final 3 weeks behave fundamentally differently from every other event category in the same window.** Without the interaction term, the model could only learn this pattern implicitly from seeing Major Sports rows alongside all other rows, which requires the model to discover the interaction on its own. Adding the explicit feature gives it a direct handle.

The practical interpretation: when pricing a Major Sports game in the final 21 days, the demand profile is specific enough to that segment that the model needs a dedicated signal, not a general "is it in the final 3 weeks" flag shared across categories.

### Why spotify_popularity Beats lastfm_listeners

Spotify's recency-weighted popularity score (0.0206 importance) outperformed Last.fm's monthly listener count (0.0097). Spotify's `popularity` is recomputed weekly based on recent plays. Last.fm's `listeners` is a slower-moving count of global monthly listeners.

The model prefers the "is this artist trending right now" signal over "how many total listeners does this artist have." For predicting near-term ticket sales velocity, recent momentum is more informative than cumulative audience size.

## What Did Not Work

**Calendar rule features:** All six calendar rules (MLK weekend, Super Bowl week, etc.) registered zero importance. Binary flags covering fewer than 10 dates in 7.5M training rows are too sparse for a tree model to find a useful split on. The leakage fix was necessary for correctness, but the replacement features did not add predictive value. 

**Team records for Major Sports:** Adding `team_wins`, `team_losses`, `team_win_pct`, and `team_streak` did not meaningfully reduce the Major Sports RMSE (31.69) surprisingly. The residual error in Major Sports is driven by event-level variance like specific matchups, playoff implications, nationally televised games, star players sitting out that team-level season records cannot capture. A 14-66 Sacramento Kings game against a 60-20 Boston Celtics team has very different demand than the W/L records alone would suggest; the *specific* matchup matters more than the season record. This requires event-level features (opponent quality, broadcast designation, playoff context) that are not available in this dataset.

**Weather:** Weather joined cleanly but registered low importance overall. The training window (Oct–Jan) has limited outdoor venue events, so the signal has few rows where it is predictive.

## Conclusion

The Part 11 sweep resolves three open questions from earlier in the project:

**1. Do time-varying external signals help?**
Yes, but selectively. Spotify popularity (recency-weighted) is the highest-importance external signal at 0.0206. Last.fm listeners contribute at 0.0097. News sentiment, Wikipedia hype, weather, and inventory velocity each appear in the feature set but with lower importance. Team records add coverage for sports events but do not solve the Major Sports accuracy problem.

**2. Does expanding the lifecycle window improve accuracy?**
Yes, and it drives most of the improvement. The `final_3_weeks_major_sports` interaction alone accounts for 48% of feature importance. The lifecycle expansion from 14 to 21 days created room to define that interaction term.

**3. Does fixing the leakage in `is_anomaly_day` hurt accuracy?**
No. The six calendar-rule replacements register zero importance, which means the original `is_anomaly_day` feature was also contributing little to the model; it was a noisy flag that happened to correlate with a handful of difficult days. Removing it and replacing with calendar rules produces a cleaner, more defensible feature set without degrading performance.

The Part 11 predictions power the [Event Explorer dashboard](https://event-explorer.streamlit.app), which shows predicted demand, urgency scores, SHAP feature contributions, and market signals for every event in the test window.
